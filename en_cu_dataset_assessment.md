# ผลตรวจชุดข้อมูลค่าไฟ EN-cu

## สรุป

จัดไฟล์ PDF ทั้ง 52 ฉบับแยกตามปี พ.ศ. เรียบร้อยแล้ว และตรวจข้อความหน้าแรกของทุกไฟล์ว่าตรงกับปีในโฟลเดอร์ทั้งหมด

ชุดข้อมูลนี้ **นำมาใช้กับ CU-EnergyBrain ได้** สำหรับวิเคราะห์การใช้พลังงานและค่าใช้จ่ายแบบรายเดือนระดับเครื่องวัด/อาคาร แต่ **ยังใช้ทำนาย Peak Demand โดยตรงไม่ได้** เพราะไม่มีข้อมูลกำลังไฟฟ้า kW, Maximum Demand หรือ load profile ราย 15 นาที

## ข้อมูลที่ดึงได้จาก PDF

เอกสารเป็นตาราง “รายละเอียดค่าใช้จ่ายพลังงานไฟฟ้าจำแนกตามเครื่องวัดภายในและหน่วยงาน” และมี text layer จึงไม่ต้อง OCR

| Field | ตัวอย่าง/ความหมาย | ใช้งาน |
|---|---|---|
| `billing_month` | มกราคม 2568 | แกนเวลารายเดือน |
| `report_date` | 13 มีนาคม 2568 | ตรวจรอบการออกรายงาน |
| `organization` | สำนักงานมหาวิทยาลัย/ชื่อหน่วยงาน | จัดกลุ่มผู้ใช้ไฟ |
| `building_code` | CEN 32, INS 12 | เชื่อม master อาคาร |
| `building_name` | เปรมบุรฉัตร, จามจุรี 5 | Dashboard/Mapping |

| `energy_units` | 15,500 หน่วย | kWh รายเดือน โดยต้องยืนยันนิยามกับเจ้าของข้อมูล |

## ใช้ทำอะไรได้

| Use case | ความพร้อม | หมายเหตุ |
|---|---|---|
| Monthly kWh baseline รายอาคาร/มิเตอร์ | ใช้ได้ | มีข้อมูลต่อเนื่อง 4 ปีเต็ม |
| Cost trend และ budget forecast | ใช้ได้หลัง clean | ต้องแยกค่า missing/ค่าติดลบให้ถูกต้อง |
| จัดอันดับอาคารใช้ไฟสูง | ใช้ได้ | ควรใช้ meter mapping ป้องกันโหลดซ้ำ |
| ตรวจ anomaly เดือนต่อเดือน | ใช้ได้ | มีทั้งหน่วย, % เปลี่ยนแปลง และหมายเหตุ |
| สร้าง meter/building master data | ใช้ได้หลัง reconcile | ชื่ออาคารและเลขมิเตอร์เปลี่ยน/ซ้ำได้ |
| ตรวจเทียบ CU Smart | ใช้ได้ | Aggregate 15-minute data เป็นรายเดือนแล้วเทียบ PDF |
| คำนวณค่าไฟอย่างเป็นทางการ | ใช้ได้บางส่วน | `amount_baht` เป็นค่าใช้จ่ายภายใน ต้องยืนยันว่าเป็นยอดบิล/สูตรใด |
| Peak Demand Forecast | ใช้ไม่ได้ลำพัง | ไม่มี kW หรือ timestamp ราย 15 นาที |
| On-Peak/Off-Peak Optimization | ใช้ไม่ได้ลำพัง | ไม่มี TOU breakdown หรือ interval load |
| แนะนำเวลาเปิดแอร์ | ใช้ไม่ได้ลำพัง | ต้องใช้ CU Smart time series + ตารางใช้งานพื้นที่ |

## ข้อควรระวังก่อนนำเข้า

1. Thai text extraction มีสระ/วรรณยุกต์เพี้ยนบางจุด เช่นตัวอักษรจาก font encoding แต่ตัวเลขอ่านได้ค่อนข้างชัด
2. ค่า `-` หมายถึงไม่มีข้อมูล ไม่ควรแปลงเป็นศูนย์โดยอัตโนมัติ
3. มีค่าผิดปกติจากต้นฉบับ เช่น `#DIV/0!` ในเปอร์เซ็นต์ และรูปแบบค่าติดลบที่เครื่องหมายอยู่ท้ายตัวเลข
4. บางแถวเป็นโหลดร่วม ไม่มีรหัสอาคารหรือมิเตอร์ และอ้างอิงเครื่องวัดอื่นในหมายเหตุ
5. จำนวนหน้าเปลี่ยนจากประมาณ 41–43 หน้าเป็น 51–52 หน้า จึงไม่ควร parser ตามเลขหน้าหรือจำนวนแถวตายตัว
6. ไฟล์ `E0ff-nov65.pdf` ใช้เลขศูนย์ในชื่อ `E0ff`; parser ควรอ่านเดือน/ปีแบบยืดหยุ่น
7. ปี 2569 มีเฉพาะเดือนมกราคม–เมษายน ณ ชุดไฟล์ปัจจุบัน

## แนวทางนำไปใช้กับ CU-EnergyBrain

1. แปลง PDF เป็นตารางด้วย coordinate-based extraction แทน regex จากข้อความทั้งหน้า
2. Normalize เลขมิเตอร์ ชื่ออาคาร ปี พ.ศ./ค.ศ. ค่าติดลบ และค่า missing
3. สร้าง mapping `internal_meter_no ↔ CU Smart node/building_id`
4. Aggregate CU Smart 15-minute kWh เป็นรายเดือน แล้วเทียบกับ `energy_units`
5. ใช้ PDF เป็น monthly cost/baseline layer และใช้ CU Smart เป็น power/peak time-series layer
6. Flag แถวที่มีโหลดร่วม, `#DIV/0!`, meter missing หรือผลรวมต่างจาก dashboard เพื่อให้คนตรวจ

## ข้อสรุป

ชุดนี้มีคุณค่ามากสำหรับ historical baseline, cost analytics, meter catalog และการตรวจความถูกต้องของ CU Smart แต่ไม่ควรใช้แทนข้อมูล Maximum Demand หรือ load profile ความละเอียดสูง การรวมสองชุดเข้าด้วยกันจะเหมาะที่สุด: **PDF บอกพลังงาน/เงินรายเดือน ส่วน CU Smart บอกพฤติกรรมโหลดและ Peak ตามเวลา**
